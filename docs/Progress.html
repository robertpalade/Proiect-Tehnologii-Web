1. Web design

We started the project by creating the main elements such as navigation bar, the home page, and the pages where charts will take place. Also, at this step we made the charts( pie chart, bar chart, line chart and map chart) using some dummy data.

2. API

At this step we created a database that stores the information available in CSV format on the official site of the fleet. Moreover, even though not that many (and quite hardcoded), we have written some functions that query the database and some scripts that prepare the information taken through the functions and wrap them in JSON format(API Endpoints).

3. Select boxes

There were created some select boxes for county, brand, national category and community category to generate the charts depending on the data which you choose from the select boxes.

4. Download buttons

At this step we implemented the download buttons for exporting data in three formats: CSV, SVG, WebP. There have been made 2 functions for each chart, one for build and the other one for updating the chart every time the user selects something else from the select boxes.

5. Updated comboboxes

For each way of data visualisation we added one or two select boxes. When there are two select boxes, the second one updates depending on the first one.

6. Code Restructuring & Hardcoding Removal

For every page of data visualisation we used to have a different JavaScript file, so we created just one for every type of chart, and a single buildChart JavaScript file. Moreover, instead of the hardcoded values given for the last 5 years inside the querying functions, we have created a script which generates automatically a single table (there were 5 before) in the database with all the information.

7. API update

We added some functions for some new features we thought of, such as the ability to see the total number of cars in a county.

8. Login and Register

At this step we added login and register pages, such that if somebody wants to enter the webpage he/ she needs to create an account firstly.

9. Added header as a separate file in all pages

We've created a new JavaScript file which contains the header because too many html pages were containing the same code.

10. Admin page

We have two types of users: normal user and administrator. The administrator has a new page from where he/she can hide the desired statistics.

11. Responsiveness

We have modified the width and the height values so that the page can also be properly seen on a tablet or smartphone.

12. Validate CSS & HTML

At this step we eliminated some warnings that we had with the CSS file and HTML pages.

13. Unit testing

We made some unit tests for API.

14. Github Administration

For keeping the code updated and managing it we used Github. Every time one of us had some useful update for the project as a whole and was sure it was ready, we would commit and push it. As a matter of fact, each of the chapters above represents, more or less, a commit one of us did.

Moreover, besides these specific commits, we also had code refactorization commits, that consist in making the code more readable.

Another important aspect of Github Administration was updating the code. After one's commit, the other one of us had the responsibility to keep up with the latest changes by pulling them.
